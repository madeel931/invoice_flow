### How It Works

```
┌─────────────────────────────────────────────────────────┐
│                    main.dart                             │
│                                                         │
│  void main() async {                                    │
│    await configureDependencies();  ◄── Initializes DI   │
│    runApp(const App());                                  │
│  }                                                      │
└─────────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│              configureDependencies()                     │
│                                                         │
│  Registers all services, repositories, data sources,    │
│  use cases, and cubits into the GetIt service locator.  │
│                                                         │
│  Generated by: injectable + build_runner                 │
└─────────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│                   GetIt Instance                        │
│                                                         │
│  sl<CustomerRepository>()  ──► CustomerRepositoryImpl   │
│  sl<InvoiceCubit>()        ──► InvoiceCubit(...)        │
│  sl<IsarDatabase>()        ──► Isar instance            │
│  ...                                                    │
└─────────────────────────────────────────────────────────┘
```

### Registration Types

| Type | Annotation | Lifecycle | Use Case |
|---|---|---|---|
| **Singleton** | `@singleton` | Created once, lives forever | Database, shared services |
| **Lazy Singleton** | `@lazySingleton` | Created on first access, lives forever | Repositories, data sources |
| **Factory** | `@injectable` | New instance on every access | Cubits, transient services |

## 2. Service Registration Flow

### Registration Order

Dependencies must be registered in the correct order to satisfy dependency chains. The typical order is:

```
1. Core Services
   └── Database (Isar instance)
   └── Logger
   └── Shared utilities

2. Data Sources
   └── CustomerLocalDataSource
   └── InvoiceLocalDataSource
   └── ItemLocalDataSource
   └── SettingsLocalDataSource
   └── BusinessProfileLocalDataSource

3. Repositories
   └── CustomerRepository ──► CustomerRepositoryImpl(dataSource)

   └── ItemRepository      ──► ItemRepositoryImpl(dataSource)
   └── SettingsRepository  ──► SettingsRepositoryImpl(dataSource)
   └── BusinessProfileRepository ──► BusinessProfileRepositoryImpl(dataSource)

4. Use Cases
   └── GetAllCustomers(repository)

5. Cubits
   └── DashboardCubit(useCases...)

   └── InvoiceCubit(useCases...)
   └── ItemCubit(useCases...)
   └── SettingsCubit(useCases...)

## 6. Future Scalability Notes

### Current DI Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| Single Isar instance | All features share one DB connection | Sufficient for V1 scope |
| No environment-based DI | Same config for dev/staging/prod | TODO: Add environment modules |
| No remote data sources | Offline-only | TODO: Add for V2 cloud sync |
| Synchronous DI init | App waits for all registrations | Acceptable for current dependency count |

### V2+ DI Additions (Planned)

When cloud sync and premium features are added (V2.0.0+), the DI map will expand:

```
New Registrations Needed:
│
├── Authentication
│   ├── FirebaseAuth instance
│   ├── AuthRepository
│   └── AuthCubit
│
├── Remote Data Sources
│   ├── CustomerRemoteDataSource (Firestore)
│   ├── InvoiceRemoteDataSource (Firestore)

├── Sync Services

├── Premium
│   ├── PurchaseRepository
│   ├── PurchaseCubit

### Multi-Module DI Strategy

As the app grows, consider organizing DI into feature modules:

```
core/di/
├── injection.dart              # Entry point
├── injection.config.dart       # Generated
├── modules/
│   ├── database_module.dart    # Isar registration
│   ├── network_module.dart     # HTTP client, API config (V2+)
│   └── analytics_module.dart   # Crashlytics, analytics (V2+)
```
